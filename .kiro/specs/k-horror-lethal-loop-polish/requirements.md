# Requirements Document

## 개요

이 Spec은 Unity 전환 버전의 다음 플레이 루프 고도화 작업을 정의한다. 목표는 별도 정산소 맵에 의존하는 흐름을 줄이고, 봉고차 내부에서 리썰 컴퍼니처럼 적재 가치와 쿼터를 즉시 확인하고 정산하는 루프를 만든 뒤, 한국형 호러 고택만의 부적/종이문 기믹과 Unity 메시 검증 자동화를 이어서 추가하는 것이다.

작업은 Human-in-the-loop 방식으로 진행한다. Task 1을 구현하고 테스트 가능 상태로 연 뒤 사용자 피드백을 반영한다. 그 뒤 Task 2, Task 3을 순차 진행한다.

## Requirement 1: 봉고차 내부 즉시 정산과 HUD 연동

**User Story:** 플레이어로서, 고택에서 훔친 물건을 봉고차 안에 내려놓으면 단말기에서 적재 금액과 쿼터를 바로 확인하고, 별도 정산소로 이동하지 않고 봉고차 안에서 정산하고 싶다.

### Acceptance Criteria

1. WHEN 봉고차 내부 `VanCargoHold`에 화물이 존재하면 THEN 봉고 단말기/모니터는 현재 적재 금액, 적재 개수, 쿼터 진행도를 실시간으로 표시해야 한다.
2. WHEN 플레이어가 봉고 허브에서 적재 화물이 있는 상태로 단말기 또는 정산 트리거를 사용하면 THEN 별도 `SettlementOffice` 맵 이동 없이 즉시 정산되어야 한다.
3. WHEN 즉시 정산이 성공하면 THEN `Quota.RecoveredValue`가 정산 금액만큼 증가하고, 봉고 내부의 물리 화물 오브젝트는 제거되어야 한다.
4. WHEN 즉시 정산이 성공하면 THEN HUD 중앙 피드백에 획득 금액이 짧게 표시되어야 한다.
5. WHEN 봉고 허브에 적재 화물이 없으면 THEN 단말기는 종가 이동 동작을 유지해야 한다.
6. WHEN 종가 맵 내부에서 봉고 복귀 단말기를 사용하면 THEN 정산하지 않고 봉고 허브로 돌아가는 동작을 유지해야 한다.

## Requirement 2: 한국적 부적/종이문 상호작용 기믹

**User Story:** 플레이어로서, 고택 내부에서 단순 폐기물 회수가 아니라 한국적인 부적과 종이문을 활용해 위험을 제어하거나 정보를 얻는 선택지를 갖고 싶다.

### Acceptance Criteria

1. WHEN 플레이어가 부적 아이템을 소지하고 봉인 가능한 문에 사용하면 THEN 해당 문에는 부적이 부착되고 일정 시간 동안 귀신형 AI의 진입이 차단되어야 한다.
2. WHEN 봉인 시간이 끝나면 THEN 문은 다시 몬스터가 통과 가능한 상태가 되어야 한다.
3. WHEN 플레이어가 종이문과 상호작용하면 THEN 문이 찢긴 상태로 바뀌고 내부 시야 확보용 틈 또는 시각 표시가 생겨야 한다.
4. WHEN 종이문이 이미 찢겨 있으면 THEN 중복 상호작용은 실패 피드백만 표시해야 한다.
5. WHEN 부적/종이문 오브젝트가 씬에 배치되면 THEN 기존 `IInteractable`, `PlayerInteractor`, `HudPresenter` 패턴으로 동작해야 한다.

## Requirement 3: Unity 메시/저사양 모드 검증 이관

**User Story:** 개발자로서, Godot 시절의 고택 메시 무결성 검사와 저사양 모드 분기 검증을 Unity 테스트로 옮겨서 대문, 담장, 장식, 바닥 유격이 회귀하지 않도록 하고 싶다.

### Acceptance Criteria

1. WHEN Unity EditMode 테스트를 실행하면 THEN 대문 하단 바닥, 대문-담장 접합부, 안채/신당 주요 진입로의 충돌 틈을 검사해야 한다.
2. WHEN 부적 또는 종이문 장식이 벽면에 배치되면 THEN 장식이 벽 안에 과도하게 박히거나 플레이어 경로를 막지 않는지 검사해야 한다.
3. WHEN Godot `project.godot`의 `low_spec_mode` 설정을 참고하면 THEN Unity 렌더링 품질 설정으로 매핑하는 가이드와 베이스 코드가 제공되어야 한다.
4. WHEN 메시 검증 테스트가 실패하면 THEN 실패 메시지는 어떤 오브젝트와 위치가 문제인지 알려야 한다.
5. WHEN 전체 EditMode 테스트와 `git diff --check`가 실행되면 THEN 새 검증 코드와 기존 플레이 루프가 모두 통과해야 한다.
