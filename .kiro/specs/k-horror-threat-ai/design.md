# Design Document

## Overview

The milestone has two connected halves: the shrine approach becomes a dark hanok-style rear compound, and threats become stage-driven AI actors rather than static reveal props.

The first implementation uses simple steering AI instead of Unity NavMesh. This is intentional: the map is generated by `KHorrorBootstrapSceneBuilder`, changes often, and currently benefits more from deterministic tests than from baked navigation data. The design keeps movement and spawn decisions isolated so NavMesh can replace the steering layer later.

## Current Baseline

The Unity migration already has:

- `ResentmentTracker` for numeric resentment and stage calculation.
- `ThreatSpawnGate` for shrine item grace time.
- `EnemyKind`, `TerritoryKind`, and `EnemyTerritoryRules`.
- `ThreatProxySpawner` that reveals a single ghost proxy and dokkaebi proxy.
- Generated rear compound and deep shrine route in `KHorrorBootstrapSceneBuilder`.
- EditMode tests for run flow, territory rules, floor continuity, map boundaries, and scene content.

The missing pieces are spawn budgets, active threat counts, real AI states, damage, and stronger route atmosphere checks.

## Components

### ThreatDirector

`ThreatDirector` is a pure C# domain service. It receives a `ThreatDirectorContext` and returns a `ThreatDirectorDecision`.

Inputs:

- Current map.
- Player territory.
- Resentment stage.
- Whether shrine grace blocks aggression.
- Active ghost count.
- Active dokkaebi count.
- Deterministic variance seed.

Outputs:

- `None`
- `CueOnly`
- `SpawnGhost`
- `SpawnDokkaebi`

The director owns budget decisions. Runtime Unity code only applies its decisions.

### ThreatStageProfile

`ThreatStageProfile` is a pure data value:

- Max active threats.
- Ghost damage.
- Dokkaebi damage.
- Detection range.
- Chase speed.
- Pattern variance.

Profiles are monotonic by stage: later stages cannot reduce damage or pressure.

### EnemyBrain

`EnemyBrain` is a small MonoBehaviour state machine:

- `Idle`
- `Lurk`
- `Stalk`
- `Chase`
- `Attack`
- `ReturnHome`
- `Despawn`

It uses vector steering toward a target or home point. Territory checks remain explicit. If the target is invalid, the brain falls back instead of crossing the gate or leaving the allowed area.

### PlayerDamageReceiver

`PlayerDamageReceiver` is a small runtime component with current health, max health, and `ApplyDamage(int amount)`. Tests can exercise damage without requiring full player movement.

### RuntimeThreatSpawner

This scene component bridges `GameLoopController`, `ThreatDirector`, and generated threat actors. It evaluates periodically, activates hidden threat actors at named spawn anchors, and assigns their target/home data.

For the first pass it can reuse generated primitive actors. Later tasks can replace actor visuals with prefabs.

### Rear Route Visuals

`KHorrorBootstrapSceneBuilder` remains the scene authoring source for now. The route from main house rear to shrine should include:

- Roofed rear gate frames.
- Low hanok walls and plaster/wood surfaces.
- Lantern pools and cold fill lights.
- Storehouse/side room silhouettes.
- Talismans and rope details.
- Tight turns that prevent direct shrine visibility.

Scene integrity tests validate named anchors and occluders rather than judging aesthetics by screenshot.

## Testing Strategy

- Pure EditMode domain tests for `ThreatDirector` budgets, territory selection, monotonic stage profiles, and variance.
- MonoBehaviour EditMode tests for `EnemyBrain` state transitions and damage.
- Scene content tests for generated route anchors, spawn anchors, and inactive initial threat actors.
- Full EditMode run after each Kiro task.

## Migration Safety

All runtime work is Unity C# under `Unity_Migration/KHorrorUnity`. Godot files remain read-only reference. Generated Unity YAML changes must be traceable to `KHorrorBootstrapSceneBuilder` or rendering assets.
