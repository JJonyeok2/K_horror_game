# Requirements Document

## 개요

이 Spec은 Unity 6 기반 `K_horror_game` 이관 프로젝트의 다음 플레이어 체감 마일스톤을 정의한다. 현재 시스템 레벨에서는 봉고차 이동, 고택 수거, 화물 적재, 즉시 정산, 기본 위협 단계가 연결되어 있다. 그러나 리썰컴퍼니식 회수 생존 게임으로 느껴지려면 적재 화물을 다시 집어 드는 루프, 몰입형 단말기와 상호작용 피드백, 한옥 내부 위협 압박, 부적과 종이문 같은 한국형 기믹이 더 구체적으로 닫혀야 한다.

작업은 Human-in-the-loop 방식으로 진행한다. 각 Task를 구현하고 Unity EditMode 테스트를 통과시킨 뒤, 사용자가 Unity에서 직접 플레이 테스트하고 피드백을 준 다음 다음 Task로 넘어간다.

## Requirement 1: 적재 화물 재픽업 루프

**User Story:** 플레이어로서, 봉고차 내부 적재 구역에 내려둔 물건을 다시 집어 들 수 있어야 한다. 그래야 잘못 둔 물건을 재배치하거나, 가치가 낮은 물건을 나중에 빼고, 정산 전 적재 상태를 직접 정리할 수 있다.

### Acceptance Criteria

1. WHEN 플레이어가 봉고차 적재 구역의 `VanCargoItem`을 카메라 레이캐스트로 바라보고 `E`를 누르면 THEN 해당 화물은 `VanCargoHold`에서 제거되고 플레이어 인벤토리에 다시 들어가야 한다.
2. WHEN 재픽업이 성공하면 THEN `VanCargoHold.CargoCount`와 `VanCargoHold.TotalCargoValue`가 즉시 감소해야 한다.
3. WHEN 재픽업이 성공하면 THEN 화물 오브젝트는 적재 슬롯에서 사라지고 플레이어의 1인칭 손 시야 가림 오브젝트로 다시 표시되어야 한다.
4. WHEN 플레이어 손 슬롯이 가득 차 있거나 무게/슬롯 조건을 만족하지 못하면 THEN 재픽업은 실패하고 화물은 적재 구역에 그대로 남아야 한다.
5. WHEN 재픽업이 실패하면 THEN 중앙 하단 HUD에 실패 이유가 표시되어야 한다.
6. WHEN 화물을 재픽업한 뒤 다시 `G`로 적재하면 THEN 같은 가치, 무게, 손 슬롯, 위협 태그 정보가 보존되어야 한다.

## Requirement 2: 봉고차 단말기와 상호작용 UI 고도화

**User Story:** 플레이어로서, 봉고차 내부 단말기를 볼 때 단순 디버그 텍스트가 아니라 실제 회수 생존 게임의 터미널처럼 적재 상태, 쿼터, 정산 가능 여부를 읽고 조작하는 느낌을 받고 싶다.

### Acceptance Criteria

1. WHEN 봉고차 단말기 UI가 갱신되면 THEN 현재 적재 가치, 화물 개수, 쿼터, 남은 목표 금액이 한 화면에 표시되어야 한다.
2. WHEN 단말기 화면이 열리거나 내용이 갱신되면 THEN 텍스트가 한 글자씩 출력되는 타이핑 효과가 적용되어야 한다.
3. WHEN 쿼터를 달성하거나 정산에 실패하면 THEN 단말기 UI는 성공/실패 상태를 색상, 짧은 애니메이션, 상태 문구로 알려야 한다.
4. WHEN 플레이어가 상호작용 가능한 대상을 바라보면 THEN 중앙 하단 프롬프트는 대상 이름과 동작을 조합해 `[E] 단말기 조작`, `[E] 화물 다시 들기`, `[E] 부적 붙이기`처럼 가변 표시되어야 한다.
5. WHEN 상호작용 대상이 사라지거나 사거리 밖으로 나가면 THEN 중앙 하단 프롬프트는 즉시 비워져야 한다.

## Requirement 3: 위협 AI 체감 압박과 사운드 오클루전

**User Story:** 플레이어로서, 위협 단계가 올랐을 때 단순히 숫자만 바뀌는 것이 아니라 한옥 내부에서 귀신이 가까이 있다는 시각/청각 압박을 느끼고 싶다.

### Acceptance Criteria

1. WHEN 귀신 또는 도깨비가 플레이어를 추적 중이면 THEN 거리, 영역, 시야 상태에 따라 발소리/울음소리/기척 이벤트가 발생해야 한다.
2. WHEN 플레이어와 AI 사이에 벽이나 종이문이 있으면 THEN 오디오 볼륨이 줄고 로우패스 필터가 적용되어 막힌 소리처럼 들려야 한다.
3. WHEN 종이문 너머에 AI가 있거나 접근 중이면 THEN 문창살 그림자, 흔들림, 긁힘 같은 시각적 단서가 발생할 수 있어야 한다.
4. WHEN 신당 최고 가치 아이템을 훔쳐 위협 단계가 최고조에 도달하면 THEN 주변 등불 깜빡임, 안개 밀도 증가, 짧은 화면 피드백 같은 환경 연출이 트리거되어야 한다.
5. WHEN 귀신은 고택 내부 영역에 있고 도깨비는 숲 영역에 있을 때 THEN 기존 영역 규칙을 깨지 않고 각 영역에 맞는 압박 이벤트만 발생해야 한다.

## Requirement 4: 부적과 종이문 기믹

**User Story:** 플레이어로서, 한국형 호러 배경에 어울리는 부적과 종이문을 사용해 몬스터 동선을 잠시 차단하거나 정보를 얻는 선택지를 가지고 싶다.

### Acceptance Criteria

1. WHEN 플레이어가 부적 아이템을 들고 봉인 가능한 종이문 또는 문틀에 상호작용하면 THEN 부적이 부착되고 일정 시간 AI 통과/파괴가 차단되어야 한다.
2. WHEN 봉인 시간이 끝나면 THEN 문은 다시 AI가 통과하거나 공격할 수 있는 상태로 돌아와야 한다.
3. WHEN 종이문이 플레이어 상호작용 또는 몬스터 공격으로 충분한 피해를 받으면 THEN 찢어진 상태로 바뀌고 시야/AI 인지 판정이 문 너머까지 확장되어야 한다.
4. WHEN 부적이 붙은 문은 THEN 몬스터가 즉시 파괴하지 못하고 다른 경로를 선택하거나 잠시 대기해야 한다.
5. WHEN 종이문 또는 부적 상호작용이 실패하면 THEN 중앙 하단 HUD에 실패 이유가 표시되어야 한다.

## Requirement 5: 작업 단위 검증과 푸시

**User Story:** 개발자로서, 각 Task가 끝날 때 테스트와 원격 푸시가 완료된 상태여야 Mac/Windows 양쪽에서 이어 작업할 수 있다.

### Acceptance Criteria

1. WHEN 각 Task 구현이 끝나면 THEN 관련 EditMode 테스트와 전체 EditMode 테스트를 실행해야 한다.
2. WHEN 테스트가 통과하면 THEN `git diff --check`를 실행해야 한다.
3. WHEN 검증이 끝나면 THEN 논리적 커밋을 만들고 `origin/unity-conversion`에 푸시해야 한다.
4. WHEN Task가 끝나면 THEN Unity 에디터를 열어 사용자가 직접 플레이 테스트할 수 있게 해야 한다.
